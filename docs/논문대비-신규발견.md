## 진짜 차별점 (실측으로 확인됨)

### A. 붙여넣은 본문이 프롬프트와 **분리 저장**된다 — `pastedContents`

논문은 *"Claude Code logs **only user input prompts** in history.jsonl"* 라고 적는다(원문) — history.jsonl 을 평면적인 "프롬프트 로그"로 본 것. 붙여넣은 것도 넓게는 user input이라 이 진술이 *틀린 건 아니다.* 우리가 더한 건 **논문이 짚지 않은 내부 구조**다: 프롬프트 한 덩어리가 아니라, 표시 칸(`display`)엔 쪽지(placeholder)만 있고 진짜 본문은 따로 흩어진다. 본문을 찾으려면 최대 3단계를 따라가야 한다.

```
~/.claude/
├─ history.jsonl                ← 한 줄 = 입력 하나
│   └─ { "display": "[Pasted text #1 +4 lines] 이러헥 떠",      ← 쪽지(placeholder)만
│        "pastedContents": { "1": { "type":"text",
│                                   "content" 또는 "contentHash" } } }   ← 본문 또는 해시
├─ paste-cache/<hash>.txt       ← contentHash 만 있으면 진짜 본문은 여기
└─ projects/<폴더>/<세션>.jsonl   ← 붙여넣은 "이미지"는 여기 레코드 안(base64)
     └─ { "type":"user", "message":{ "content":[
            { "type":"text",  "text":"[Image #1] …" },              ← 쪽지
            { "type":"image", "source":{ "type":"base64",
                                         "data":"iVBORw0KGgo…" }}]}}  ← PNG inline
```

- **본문이 항상 `content` 에 있는 건 아니다**(실측): `pastedContents` 항목 2239개 중 `content` 직접 보유 892개 / `contentHash` 만 1347개 → 후자는 본문이 `~/.claude/paste-cache/<hash>.txt` 에 따로 있음(369개 캐시 파일 확인).
- **`pastedContents` 는 텍스트 전용**(2239개 전부 `type:text`). 붙여넣은 **이미지**는 history.jsonl 안 거치고 **projects transcript 레코드 안에 base64로 inline** 저장된다: `message.content[]` 의 `type:image` 파트 → `source.data`(base64 PNG), `imagePasteIds` 가 `[Image #N]` 쪽지와 매핑. base64 디코드하면 원본 PNG 복원(실측: 1100×1318 등 정상 추출).
- 포렌식 의미: 프롬프트(`display`)만 읽으면 붙여넣은 내용(=유출 후보)을 통째로 놓친다. `display → pastedContents → paste-cache` 사슬을 끝까지 따라가야 본문이 보인다.
- 실측: `history.jsonl` 10801줄 중 **2153줄**이 `pastedContents` 보유.
- 검증 명령:
  ```bash
  python3 -c "import json,os; rows=[json.loads(l) for l in open(os.path.expanduser('~/.claude/history.jsonl'),encoding='utf-8',errors='ignore') if l.strip()]; pc=[r for r in rows if r.get('pastedContents')]; items=[v for r in pc for v in r['pastedContents'].values()]; print('총',len(rows),'/ pasted줄',len(pc),'/ content직접',sum(1 for v in items if v.get('content')),'/ hash만',sum(1 for v in items if not v.get('content') and v.get('contentHash')))"
  ```

### B. 논문이 **열거하지 않은** 증거 레코드 타입 — `projects/*.jsonl`

논문은 Claude `projects/` 를 한 줄로만 적는다: *"session indices and associated data are distributed across the projects/ directory"* — **레코드 타입을 하나도 열거하지 않는다**(논문 원문 대조: 아래 타입 전부 0회 언급). 즉 논문은 `projects/` 를 불투명 상자로 두고, 우리는 그 안의 포렌식 증거 타입을 매핑한다.

> 정직: "이 타입들이 2.1.72엔 *없었다*"는 주장은 안 한다(옛 버전 미설치 → 증명 불가). 확실한 건 **논문이 다루지 않았다**는 것. 일부(`thinking` 등)는 과거에도 있었을 수 있다.

구조 주의: `.jsonl` 한 줄 = JSON 레코드 하나, 줄마다 `type` 필드로 종류 구분. 아래 대부분은 **줄의 `type` 값**이지만, `toolUseResult` 은 type이 아니라 `user`/`assistant` 레코드 **안의 필드(key)** 다.

```
~/.claude/projects/
└─ <작업폴더 인코딩>/              ← 디렉토리 (cwd별 자동 분리 → 다른 프로젝트와 안 섞임)
    └─ <세션UUID>.jsonl          ← 파일 (세션 1개, 한 줄 = 레코드 1개)
         ├─ { "type":"user", "toolUseResult":{ "file":{ "filePath","content" }}}  ← 읽은 파일 원문(필드)
         ├─ { "type":"permission-mode",       "permissionMode":"bypassPermissions" }
         ├─ { "type":"file-history-snapshot", "snapshot": … }
         ├─ { "type":"agent-name",            "agentName": … }   ← isSidechain 필드 동반
         └─ { "type":"thinking", … }
```

| 이름                                   | 정체            | 무엇                                        | 왜 증거               |
| ------------------------------------ | ------------- | ----------------------------------------- | ------------------ |
| `toolUseResult`                      | 레코드 안 **필드**  | 도구 실행 결과 — 읽은 파일 경로+원문(`.file.content`)   | 무슨 비밀/소스를 봤는지      |
| `file-history-snapshot` (7,179)      | `type` 값       | 편집 전 파일 백업(`snapshot`)                     | 원본 상태 복원           |
| `permission-mode` (11,990)           | `type` 값       | 위험모드(`bypassPermissions`) 여부              | 공격 가능성의 핵심 신호      |
| `agent-name` (5,343) / `isSidechain` | `type` 값/필드    | 어느 서브에이전트가 한 행동인지                         | 행위 귀속(attribution) |
| `thinking` (37,659)                  | `type` 값       | 에이전트 내부 추론                                | 의도·계획              |

- 검증 명령:
  ```bash
  grep -rhoE '"type":"(toolUseResult|file-history-snapshot|permission-mode|agent-name|thinking)"' ~/.claude/projects 2>/dev/null | sort | uniq -c
  ```

### C. 논문 Table 2가 **열거하지 않은** 디렉토리 (의미 있는 것만)

논문 Table 2가 `~/.claude` 경로로 나열한 건 6개뿐: `.claude.json` · `.credentials.json` · `history.jsonl` · `plugins/` · `projects/` · `settings.json`. 실제 머신엔 디렉토리가 20개+ 있다. 그중 **논문이 안 적었고(0회) + 포렌식 의미 있는 것**만:

`file-history/` (편집 전 파일 백업) · `paste-cache/` (붙여넣기 본문) · `shell-snapshots/` (셸 상태) · `plans/` (계획)

> 정직(반박 방지) — 뺀 것들:
> - `plugins/` : 논문 Table 2에 이미 있음.
> - `hooks/` : 논문이 §6.2 "Hooks 주입 RCE" 시나리오로 크게 다룸 → '새 발견' 아님.
> - `tasks/` : 논문엔 "coding tasks"라는 *단어*로만 나옴, `~/.claude/tasks/` 디렉토리 근거 아님.
> - `cache/`·`downloads/`·`image-cache/` : 단순 캐시(포렌식 가치 낮음).

- 검증 명령:
  ```bash
  ls -d ~/.claude/{file-history,paste-cache,shell-snapshots,plans}/ 2>/dev/null
  ```

## 검증했지만 차별점이 아닌 것 (정직 — 같은 착오 반복 방지)

이 항목들은 처음엔 "신규 발견"으로 적었으나 **실데이터로 확인하니 틀렸거나 약했다.** 발표에 차별점으로 넣지 말 것.

| 항목 | 처음 생각 | 실측 결과 | 판정 |
|---|---|---|---|
| 버전 드리프트 (2.1.72→2.1.178) | 신규 발견 | 시간 지나 버전 오른 것뿐 | **배경**일 뿐, 기여 아님 |
| Codex가 Fernet 안 씀(평문화) | 평문으로 읽힘 | 222개 세션 중 **193개에 `encrypted_content:"gAAAAA"`(Fernet) 여전히 존재**. 추론은 아직 암호화, 겉(프롬프트·cwd·모델)만 평문 | **거짓** — 삭제 |
| Windows+WSL2 이중 저장소 | 플랫폼 차별점 | 윈도우+WSL 둘 다 설치 시 생기는 **설치 선택**, 파일 형식 동일 | **차별점 아님** |
| `agents/`·`teams/` = 멀티에이전트 이미 존재 | 신규 아티팩트 | 두 폴더 **모두 비어 있음(empty)**. 진짜 서브에이전트 흔적은 폴더가 아니라 §B의 `agent-name`/`isSidechain` 에 있음 | **거짓** — §B에 흡수 |

> 교훈: 디렉토리·파일이 "존재"하는 것과 "증거를 담는" 것은 다르다. 빈 폴더는 증거가 아니다. 항상 **안의 내용**을 확인하고 적는다.
