# 정확도 검증 시나리오 (발표용)

도구의 핵심 판정 정확도를 **자동으로** 측정한다. 발표 데모용이므로 베이스라인 대조·신뢰구간 같은 논문급 엄밀도는 생략하고, "정답을 아는 합성 기록을 넣고 → 도구 출력과 1:1 비교 → 정확도" 만 돌린다.

검증 대상 3가지 (각 100건):
1. **주체 식별** — 한 행위의 주체가 사용자(A)냐 에이전트(B)냐.
2. **접근 파일 판정** — 어떤 파일에 접근(read/write)했나 + 무접근 파일을 잘못 잡지 않나.
3. **MCP 호출 탐지** — 외부 MCP 툴 호출을 잡고 server/tool을 맞히나.

> 단일 진실원천: 정답(ground-truth)은 `event-schema.md`의 `actor`·`action`·`target` 형식 그대로 쓴다. 도구 출력 Event와 **필드 단위 1:1 비교**. 새 평가 필드 만들지 말 것.

---

## 0. 공통 원리 — 정답을 아는 합성 transcript

도구 입력 = Claude Code 기록 트리. `ClaudeSource(root)`가 읽는 레이아웃:

```
<root>/
├─ history.jsonl                      # (선택) 사용자 프롬프트 히스토리
└─ projects/<proj>/<session>.jsonl    # 세션 transcript (rglob "*.jsonl")
```

검증은 **이 트리를 합성으로 만든다.** 각 jsonl 줄이 한 레코드고, 파서가 그걸 Event로 바꾼다. 우리가 레코드를 만들 때 정답 라벨을 같이 적어두면, 도구가 뱉은 Event와 비교만 하면 된다.

### 레코드 → Event 매핑 (parser.py 실제 규칙)

| 만들 레코드 (jsonl 한 줄) | 도구가 내는 Event | 정답 |
|---|---|---|
| `{"type":"user","message":{"content":[{"type":"text","text":"..."}]}}` | actor=**user**, action=**prompt** | 사용자(A) |
| `{"type":"user","message":{"content":[{"type":"image",...}]}}` | actor=**user**, action=**paste** | 사용자(A) |
| `{"type":"assistant","message":{"content":[{"type":"tool_use","name":"Bash","input":{"command":"..."}}]}}` | actor=**agent**, action=**bash**, target=명령 | 에이전트(B) |
| `{"type":"assistant",...,"tool_use","name":"Write","input":{"file_path":"/p"}}` | actor=**agent**, action=**write**, target=/p | 에이전트(B) |
| `{"type":"assistant",...,"tool_use","name":"mcp__notion__search","input":{...}}` | actor=**agent**, action=**mcp**, target=mcp__notion__search | 에이전트(B) |
| `{"type":"assistant",...,"text":"..."}` | actor=**agent**, action=**response** | 에이전트(B) |
| `{"toolUseResult":{"file":{"filePath":"/p","content":"..."}}}` | actor=**agent**, action=**read**, target=/p | 에이전트(B) |

각 레코드에 `"timestamp"`(ISO8601 UTC), 같은 `"sessionId"`를 넣는다. 모든 Event는 `source.file`(그 jsonl 경로)·`source.line`(줄 번호)을 자동으로 단다 → 출처 인용이 그대로 증거가 된다.

### 도구 실행 (검증 진입점)

```python
from clfx.sources.claude import ClaudeSource
from clfx.parser import parse_source
from clfx.analyze.mcp import mcp_summary

src = ClaudeSource(root)               # 합성 트리
events = list(parse_source(src))       # ← 도구가 복원한 Event 전수 (주체·접근 판정의 결과)
mcp = mcp_summary([root], events)      # ← MCP 탐지 결과(usage)
```

채점기는 `events`(+`mcp`)를 정답 라벨과 비교한다. 추가 읽기·재집계 없음 — 도구가 내놓은 그대로.

## 1. 시나리오 1 — 주체 식별 정확도 (100건)

**질문**: 각 행위의 actor를 A(user)/B(agent)로 맞히나. 특히 **④ 왜곡 케이스**(에이전트가 작성했지만 디스크 소유자는 사람 계정)에서도 B로 잡나.

**생성 (100 레코드)**: 사용자 행위·에이전트 행위를 섞어 100개. 권장 구성(정답이 한쪽에 쏠리지 않게):
- user prompt 30, user paste(image) 10  → 정답 A 40
- assistant bash 20, assistant write 20, assistant response 20 → 정답 B 60
- 그중 write 20개는 **왜곡 케이스**로 표시: 같은 파일을 실제 디스크(임시폴더)에도 만들어 두면(소유자=실행계정=사람), owner만 보면 A로 오인되는 상황. 도구는 transcript actor로 B를 내야 정답.

각 레코드 생성 시 `gt_actor ∈ {user, agent}`를 같이 기록.

**측정**: 도구 Event마다 `(source.file, source.line)`로 정답 레코드와 매칭 → `event.actor == gt_actor` 비율.
- 주지표: **정확도 = 맞은 수 / 100**.
- 보조: 2×2 혼동행렬(A→A, A→B, B→A, B→B). 특히 **왜곡 20건 중 B로 맞힌 수**를 따로 표기(헤드라인).

**합격기준(데모)**: 전체 정확도 100% 기대(통제 합성이라 결정적). 왜곡 20/20. 하나라도 틀리면 파서/귀속 버그 → 원인 보고.

---

## 2. 시나리오 2 — 접근 파일 판정 정확도 (100건)

**질문**: 실제 접근(read/write)한 파일을 빠짐없이 잡고(누락 0), 접근 안 한 파일(decoy)을 잘못 잡지 않나(오탐 0). action(read/write)도 맞나.

**생성 (100 파일 기준)**:
- 접근 파일 70: read 35(toolUseResult.file), write 35(Write tool_use). 각 `gt={path, action}`.
- decoy 30: 임시폴더에 **파일은 만들되 transcript엔 등장 안 시킴** → 도구가 잡으면 오탐.
- 경로는 임시폴더 하위 실제 경로로(접근 파일 판정·정규화까지 태움).

**측정**: 도구 Event 중 action∈{read,write}의 `target` 집합 vs 정답 접근파일 집합.
- 누락(정답인데 도구가 안 냄) = recall 손실 → **0이어야 함**(무손실 제약).
- 오탐(decoy를 냄) = precision 손실 → **0이어야 함**.
- action 일치율(맞춘 파일 중 read/write 정답 비율).
- 주지표: **정확도 = (정답집합 == 도구집합) 인가 + action 일치율**. 표기: 누락 N, 오탐 M, action 정확도 %.

**합격기준(데모)**: 누락 0, 오탐 0, action 100%.

---

## 3. 시나리오 3 — MCP 호출 탐지 정확도 (100건)

**질문**: 외부 MCP 툴 호출을 잡고 server/tool을 맞히나. 일반 툴(Bash/Write 등)을 MCP로 오인하지 않나.

**생성 (100 tool_use 기준)**:
- MCP 호출 60: `name="mcp__<server>__<tool>"` (server/tool 여러 종 — notion/playwright/ghidra 등 골고루). 각 `gt={server, tool}`.
- 비-MCP 40: Bash/Write/Read 등 일반 tool_use(decoy) → MCP로 잡으면 오탐.

**측정**: 도구 Event 중 `action=="mcp"` 의 `target`(=풀네임)을 `mcp__server__tool`로 분해(`_split_target`과 동일 규칙) → 정답 server/tool과 비교. `mcp_summary(...).usage` 의 server/tool 집계도 교차 확인.
- 탐지 누락(MCP인데 못 잡음) 0, 오탐(비-MCP를 MCP로) 0.
- server/tool 식별 일치율.
- 주지표: **정확도 = MCP 탐지 정오 + server/tool 일치율**.

**합격기준(데모)**: 탐지 누락 0, 오탐 0, server/tool 100%.

## 4. 자동화 하네스 (설계 — 다음 단계 구현)

`scripts/eval/` 한 폴더로 끝낸다. stdlib만. read-only(증거 트리는 임시폴더, 끝나면 폐기).

```
scripts/eval/
├─ gen.py     # 시나리오별 합성 트리 + 정답 JSON 생성 (각 100건). 결정적 시드.
├─ run.py     # ClaudeSource→parse_source→(mcp_summary) 실행, 채점, 표 출력
└─ out/       # 생성된 임시 트리·정답·결과 리포트 (gitignore)
```

흐름:
1. `gen.py` → `out/<scenario>/projects/.../session.jsonl` + `out/<scenario>/truth.json`(라벨 배열). 타임스탬프·시드 고정(결정적).
2. `run.py` → 각 시나리오 도구 실행 → Event를 `(source.file, source.line)` 또는 `target`으로 정답과 매칭 → 정확도/혼동/누락/오탐 계산.
3. 출력(콘솔 표 + `out/report.md`):

```
시나리오            건수   정확도   비고
주체 식별           100    100.0%   왜곡 20/20
접근 파일 판정       100    100.0%   누락 0 · 오탐 0 · action 100%
MCP 호출 탐지        100    100.0%   누락 0 · 오탐 0 · server/tool 100%
```

실행:
```bash
python scripts/eval/gen.py            # 합성 데이터셋 생성
python scripts/eval/run.py            # 검증 + 리포트
```

**주의(완전성·결정성)**: 합성 트리만 검증 대상. 실데이터 `~/.claude`는 건드리지 않는다(read-only). 생성은 시드 고정으로 재실행 동일. 도구는 어떤 행위도 skip/sample 없이 전수 복원해야 정답과 일치한다(무손실 제약이 곧 합격조건).

> 이 문서는 "어떻게 검증하나"의 설계다. `scripts/eval/` 실제 구현은 별도 작업으로 진행(이 문서대로).
