# CLI 에이전트 포렌식 도구 — 설계

## 한 줄 요약

개발자가 쓰는 AI 코딩 에이전트(Claude Code · Codex · Gemini CLI)가 디스크에 남긴 기록을 **파싱하고, 분석하고, 자연어로 물어볼 수 있는** 도구.

## 왜 만드나

**① 에이전트 급증, 그리고 실제 유출 사건.** 개발자가 AI 코딩 에이전트를 널리 쓰면서, 사내 기밀이 외부 AI 서버로 나간 사건이 실제로 일어났다 — 삼성 직원이 소스코드를 ChatGPT에 입력(2023), 악성 패키지가 에이전트를 무기화해 시크릿을 빼낸 Nx 사건(2,180계정, 2025), 프롬프트 주입으로 허용된 `api.anthropic.com` 엔드포인트를 통해 데이터를 유출한 Claudy Day·Rehberger 취약점(2025–2026).

**② 막는 것(예방)은 불완전하다.** 예방형 DLP는 브라우저·엔드포인트 중심이라 CLI 에이전트엔 구멍이 크다 — 허용된 API 엔드포인트로 나가면 시그니처에 안 걸리고, 프롬프트 주입·직접 DNS·인코딩으로 우회되며, 개발자의 WSL·컨테이너 환경이 회사 통제를 흔히 비껴간다(위 사건들이 그 증거). 코딩 에이전트 보안 제품이 나오고는 있지만 대부분 *예방·차단*이다.

**③ 그런데 사고 난 뒤 조사할 도구가 없다.** 에이전트는 시킨 일·읽은 파일·붙여넣은 내용을 전부 디스크에 기록한다(`~/.claude/projects/*.jsonl`, `history.jsonl`, `paste-cache/` 등). 사고 후 이걸 분석하면 핵심 증거가 되는데, 지금은 사람이 수작업으로 jsonl을 뒤지거나 프록시 로그(내용은 안 보임)만 본다. **우리는 예방 도구가 아니라, 사고 후 이 기록을 파싱·분석하는 사후(post-incident) 포렌식 도구다.**

## 기존 연구와 우리

선행 논문(Kim & Jeong, *From assistant to accomplice*, SSRN 6725750)은 CLI 에이전트가 남기는 아티팩트를 **사람이 수동으로 분석하는 프레임워크**를 제안했다. 우리는 그 위에 두 가지를 더한다.

1. **실제로 돌아가는 자동화 도구** — 논문은 개념과 수동 분석에 그친다. 우리는 파싱→분석→질의를 자동으로 하는 도구를 만든다.
2. **논문이 다루지 않은 것까지 파싱·분석** — 논문은 `projects/` 를 불투명하게 두지만(경로만 적고 내부는 안 깠다), 그 안엔 붙여넣은 본문(`pastedContents`→`paste-cache`), 편집 전 파일 백업(`file-history`), 서브에이전트(`agent-name`) 같은 증거가 있다 (`논문대비-신규발견.md` 참고). 여기에 시크릿/개인정보 자동 탐지를 더한다.

> 논문은 Ubuntu에서 실험했지만, 에이전트 기록 파일은 OS와 상관없이 같은 형식이라 그건 차별점이 아니다. 우리 차별점은 *자동화* 와 *추가로 분석하는 내용* 이다.

## 무엇을 만드나 — 세 단계

1. **파싱** — `~/.claude/projects/*.jsonl` 같은 에이전트 기록을 읽어 구조화한다 (프롬프트, 읽은 파일, 실행한 명령, 시각 등).
2. **분석** — 구조화된 기록에서 의미를 뽑는다: 무엇을 언제 했는지 타임라인, 사람이 직접 입력했는지 에이전트가 알아서 했는지, 시크릿·개인정보가 새어나갔는지.
3. **질의** — 분석 결과에 자연어로 묻는다 ("누가 .env 읽었어?", "유출된 비밀 뭐야?"). 로컬 AI가 답하되, 모든 답은 원본 기록(파일·줄 번호)으로 되짚을 수 있게 한다.

## 파싱 대상

- **Claude Code** — `~/.claude/projects/*.jsonl`, `history.jsonl`, `file-history/`, `paste-cache/`, 설정·자격증명
- **Codex CLI** — `~/.codex/sessions/`, 설정
- **Gemini CLI** — `~/.gemini/tmp/.../chats/`, 설정

자세한 경로와 논문 대비 새로 발견한 것은 `논문대비-신규발견.md`.

## 출력 형태

파싱·분석 결과는 일관된 레코드 하나로 정리한다. 형식은 `event-schema.md` (핵심 필드만).

## 분석이 쓸모있는 상황 (시나리오)

**상황**: 한 직원이 기밀 유출로 의심받는다. 회사 내부 포렌식팀(또는 용역 조사업체)이 그의 에이전트 기록을 조사한다.

기밀이 **외부 제3자 AI 서버**(Anthropic/OpenAI/Google)로 나간 것은 두 경우 모두 같다 — 회사 정보보안 규정 위반은 양쪽 다 성립(경쟁사 여부와 무관). 도구가 가르는 건 "유출 여부"가 아니라 **그 행위를 누가 했나(행위 주체)**다:

- **A. 사용자 직접 입력** — 직원이 기밀을 직접 Claude 입력창에 **붙여넣음**. 증거 = `history.jsonl`/`paste-cache` 의 붙여넣기 본문 (`actor:user`, `action:paste`). → 사람이 직접 규정을 위반한 객관적 증거.
- **B. 에이전트 자율 접근** — 위험모드(`--dangerously-skip-permissions`)나 악성 패키지(Nx식)가 에이전트를 시켜 **직원 입력 없이** `.env`를 읽음. 증거 = `projects/*.jsonl` 의 `toolUseResult`+`permission-mode` (`actor:agent`, `action:read`). → 사람이 직접 한 게 아니란 증거(위험모드를 켜둔 과실이나 악성 패키지 피해 여부는 별개).

도구의 역할 = **그 행위를 누가 했나(`actor`)를 증거로 밝히는 것**까지다. 누가 했나는 책임 소재의 출발점이지만, 징계·처벌 판단은 도구 밖(인사·법무·수사)에서 한다 — 도구는 무엇이·누가·언제를 원본 줄로 추적해 **판단 근거**를 줄 뿐이다.

## 평가

정답을 아는 **통제 데이터셋**을 만들어 (`스테이징-데이터셋-가이드.md`), 도구가 사건을 맞게 재구성하는지, 시크릿을 몇 개나 잡아내는지(정확도)를 측정한다.

## 지금 안 하는 것 (나중 레이어)

핵심 동작(파싱·분석·질의)에는 필요 없으므로 뒤로 미룬다.

- 삭제된 기록 복구
- 디스크 이미지(ext4.vhdx)에서 추출 — 그냥 파일을 직접 읽으면 됨
- Windows 이벤트로그와 교차

메모리 포렌식·네트워크 캡처는 범위 밖(디스크 기록만 다룬다). 실시간 차단도 안 한다(사고 후 분석 도구다).
